# TrendRadar 项目评估报告

> **日期**：2026-04-21
> **项目地址**：https://github.com/sansan0/TrendRadar
> **评估目的**：评估是否可以接入品猹平台，用于每日资讯获取

---

## 一、项目概览

TrendRadar（v6.6.1，GPL-3.0）是一个**全网热点新闻聚合与推送工具**，核心能力是将多个中文平台的实时热榜汇总，经关键词过滤或 AI 筛选后，以多渠道（飞书/钉钉/Telegram/邮件等）推送给用户。

**技术栈**：Python 3.12 + requests + feedparser + LiteLLM + fastmcp + SQLite

**核心模块**：

```
trendradar/
├── __main__.py        # 主入口，NewsAnalyzer 核心控制器
├── context.py         # AppContext：全局上下文与状态管理
├── core/              # 业务逻辑核心（配置、分析、调度、关键词匹配）
├── crawler/           # 数据抓取（NewsNow API + RSS/Atom）
├── ai/                # AI 分析（趋势分析、智能筛选、翻译）
├── notification/      # 多渠道推送（飞书/钉钉/Telegram/Email 等）
├── storage/           # 存储抽象（本地 SQLite + S3 远程）
└── report/            # HTML 报告渲染
mcp_server/            # MCP AI 对话服务（独立进程，17 个工具）
```

## 二、数据源

| 类型 | 数量 | 具体平台 |
|------|------|----------|
| **热榜平台** | 11个 | 今日头条、百度热搜、华尔街见闻、澎湃新闻、**Bilibili 热搜**、财联社、凤凰网、贴吧、微博、抖音、知乎 |
| **RSS 订阅** | 3个默认 | Hacker News、阮一峰博客、雅虎财经（可自由添加任意 RSS） |

**关键发现**：
- 所有热榜数据**统一通过第三方 NewsNow API** 获取（`https://newsnow.busiyi.world/api/s?id={platform_id}`），不直接爬取各平台
- 有 **Bilibili 热搜**，但只返回话题词条，不含具体视频 URL
- **无 YouTube 数据源**
- 科技/AI 热点主要靠 Hacker News RSS + 知乎/头条的泛科技内容

---

## 三、数据流

```
定时触发（每30分钟，GitHub Actions / Docker cron）
  → 抓取 11 个平台热榜（NewsNow API）
  → 抓取 RSS 订阅源（feedparser）
  → 存入 SQLite（按天归档）
  → 关键词过滤 或 AI 智能筛选（LiteLLM）
  → AI 趋势分析（可选）
  → AI 翻译（可选）
  → 生成 HTML 报告 + 多渠道推送
```

## 四、输出格式

| 输出形式 | 说明 |
|---------|------|
| Markdown 消息 | 主推送格式，适配飞书/钉钉/企业微信/Telegram/Slack |
| HTML 报告 | 完整网页版，支持暗色模式、搜索、Tab 分组 |

## 五、与品猹的接入评估

### 5.1 接入方式（推荐）

**独立 Docker 容器 + MCP Server HTTP 调用**。在品猹的 `docker-compose.yml` 中新增两个容器：
- `trendradar`：定时抓取 + 数据存储
- `trendradar-mcp`：MCP Server（端口 3333），提供 17 个查询工具

品猹后端通过 HTTP 调用 MCP Server 的 `/mcp` 端点获取热点数据。

### 5.2 接入成本

| 阶段 | 工作内容 | 预估时间 |
|------|----------|----------|
| 部署调试 | docker-compose 加容器，打通 volume | 0.5 天 |
| 后端对接 | FastAPI 写 MCP 调用代理层，注入热点数据 | 2-3 天 |
| 前端展示 | 品猹界面增加"相关热点"模块 | 1-2 天 |
| **合计** | | **约 1 周** |

### 5.3 最有价值的场景

> 用户分析某个视频时，自动检索该话题近期的热榜趋势作为背景上下文注入 LLM，丰富摘要和问答质量。

## 六、风险评估

| 级别 | 风险项 | 说明 |
|------|--------|------|
| **高** | 数据源单点依赖 | 所有热榜依赖 NewsNow 非官方 API，作者明确说明靠"善意和信任"运作，随时可能失效 |
| **高** | 反爬封禁 | 知乎/微博/头条等平台的反爬压力由 NewsNow 承担，大量调用可能加速触发限速 |
| **中** | GPL-3.0 许可证 | 若品猹作为闭源商业产品集成，存在合规风险，需法务确认 |
| **中** | 个人维护项目 | 无 SLA，更新频繁但长期稳定性无保证 |
| **低** | 依赖冲突 | Docker 容器隔离，风险为零。LiteLLM 版本与品猹兼容 |

---

## 七、结论

**可行性：可以接入，但有明显局限。**

**优势**：
- 接入成本低（约 1 周），Docker 部署简单
- 覆盖 11 个中文热榜 + RSS，对"内容遴选"模块有直接价值
- MCP Server 提供结构化查询能力，可用于上下文增强

**不足**：
- 核心数据源是第三方非官方 API，稳定性无保障
- 无 YouTube 数据源，对品猹核心场景帮助有限
- Bilibili 只有热搜词条，无视频级数据
- GPL-3.0 许可证对商业使用有约束

**最终建议**：
不作为品猹内容遴选的核心方案，推荐采用 RSSHub 自建方案（见 `PRD-内容遴选服务.md`）。TrendRadar 可作为备选的热榜数据补充源。
